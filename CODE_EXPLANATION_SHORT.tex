\chapter{Introduction \& Architecture}
% ==========================================================================

\section{What the Project Does}

A \textbf{Genetic Algorithm (GA)} automatically discovers the best \textbf{system
prompt} --- a paragraph of English instructions --- for a Claude model that generates
Python code.  Each prompt is a \emph{chromosome}; fitness is the average fraction of
test cases the LLM-generated code passes on 100 APPS coding problems.  Over 20
generations of selection, crossover, and mutation the GA evolves a prompt that solves
more problems correctly.

\begin{concept}{Key idea}
Manual prompt engineering is trial-and-error.  This project automates it: the GA
\emph{searches} all possible English instruction paragraphs, using test-case pass
rates as the fitness signal.
\end{concept}

\section{Two-Model Architecture}

\begin{center}
\begin{tabular}{@{}llll@{}}
\toprule
\textbf{Role} & \textbf{Model} & \textbf{Temp.} & \textbf{Purpose} \\
\midrule
Target (subject) & \fn{claude-haiku-4-5-20251001} & 0.0 & Generate code; fitness eval \\

Temperature 0.0 on the target model ensures the same prompt always produces the same
code on the same problem, making fitness scores comparable across generations.
Temperature 0.7 on the optimizer introduces creativity for blending and mutating prompts.

\section{Master Call-Flow (one generation)}

\begin{lstlisting}[style=chain]
main.main()
  |-- dataset.load_problems(APPS_TRAIN)       -> list[dict]  (~2346 problems)
  |-- dataset.split(problems, 100, 200, 42)   -> (evolution[100], holdout[200], rest)
  |
  +-- ga.run(SEED_PROMPTS, evolution_pool, on_generation=save_checkpoint)
        |
        | [GEN 0] for each of 15 seeds:
        |   compute_fitness(ind, problems)          [100 API calls per seed]
        |     -> llm.generate_code(prompt, q, fn)   -> raw LLM text
        |       -> llm._call("fitness", Haiku, T=0) -> resp.content[0].text
        |           -> tracker.record(tokens)
        |     -> evaluator.extract_code(response)   -> code string
        |     -> evaluator.run_tests(code, ...)      -> float [0,1]
        |
        | [GENS 1..20] for each of 13 new offspring:
        |   tournament_select(population)  x2        -> parent_a, parent_b
        |   breed(parent_a, parent_b)
        |     -> llm.crossover(a, b)   [80%]         -> merged prompt
        |     -> llm.mutate_inject(p)  [25%]         -> prompt + new strategy
        |     -> llm.mutate_delete(p)  [15%]         -> prompt - weak sentence
        |     -> llm.mutate_rephrase(p)[10%]         -> prompt (rephrased)
        |     -> Individual(prompt, uid, parents, ops)  [fitness = -1]
        |   compute_fitness(child, problems)          [100 API calls per child]
        |   save_checkpoint(run_dir, gen, population) -> generation_NN.json
        |
        +-- returns (best_Individual, history[21 dicts])
  |
  |-- evaluate_on_holdout(best.prompt, holdout[200])  [200 API calls]
  +-- tracker.flush(extra)   -> Code/Costs/cost_TIMESTAMP.json
\end{lstlisting}

\begin{concept}{API call scale}
One full run makes approximately:\\
$15 \times 100 = 1{,}500$ fitness calls (gen 0) +
$20 \times 13 \times 100 = 26{,}000$ (gens 1--20) +
$200$ (holdout) = \textbf{~27,700 Haiku calls} + ~800 Sonnet calls for operators.
\end{concept}

\section{Key Constants Explained}

\begin{lstlisting}[style=py, caption={config.py (abridged)}]
from pathlib import Path
from dotenv import load_dotenv
load_dotenv()  # reads .env for ANTHROPIC_API_KEY

ROOT         = Path(__file__).parent.parent     # Project/
APPS_TRAIN   = ROOT / "Dataset Examples/APPS/train"
RESULTS_DIR  = Path(__file__).parent / "results"   # Code/results/

EVOLUTION_SIZE = 100   # problems used as fitness benchmark
HOLDOUT_SIZE   = 200   # problems reserved for final test
RANDOM_SEED    = 42

POPULATION_SIZE   = 15
GENERATIONS       = 20
ELITISM_COUNT     = 2   # top-2 survive unchanged every generation
TOURNAMENT_SIZE   = 3   # pick best of 3 random draws for each parent

CROSSOVER_RATE    = 0.8    # probability crossover fires per offspring
MUT_INJECT_PROB   = 0.25   # probability inject mutation fires (independent)
MUT_DELETE_PROB   = 0.15
MUT_REPHRASE_PROB = 0.10

TARGET_MODEL       = "claude-haiku-4-5-20251001"
TARGET_TEMPERATURE = 0.0      # deterministic; repeatable fitness
OPTIMIZER_MODEL       = "claude-sonnet-4-6"
OPTIMIZER_TEMPERATURE = 0.7   # creative; varied crossover/mutation
MAX_CODE_TOKENS     = 1024
MAX_OPERATOR_TOKENS = 600
EXEC_TIMEOUT = 5   # seconds before subprocess is killed
\end{lstlisting}

\begin{concept}{Elitism prevents regression}
\fn{ELITISM\_COUNT=2} guarantees the top 2 prompts always pass unchanged into the next
generation.  Without elitism, an excellent prompt found in generation 5 could be lost
if all its offspring happen to score worse.
\end{concept}

\begin{concept}{Mutations are independent}
Each of the three mutation probabilities is checked with a separate \fn{rng.random()}
call, so all three can fire on the same offspring (probability $0.25 \times 0.15 \times
0.10 = 0.00375$), or none can fire ($0.75 \times 0.85 \times 0.90 = 0.574$).
This gives $2^3 = 8$ possible mutation combinations per offspring.
\end{concept}

\chapter{\pyf{seeds.py} --- Initial Population}
% ==========================================================================

\pyf{seeds.py} defines \fn{SEED\_PROMPTS}: a Python \texttt{list[str]} of 15 diverse
instruction paragraphs.  Each is 2--4 sentences emphasising a different coding strategy:

\begin{center}
\begin{tabular}{@{}rll@{}}
\toprule
\# & Strategy & Core emphasis \\
\midrule
1  & Structured analysis     & Identify types and edge cases before coding \\
2  & Competitive programmer  & Find the algorithm pattern first \\
3  & Return-type focus       & Match exact return type and format \\
4  & Step-by-step reasoning  & Four-step process: understand, identify, handle, code \\
5  & Example-driven          & Trace examples manually before implementing \\
6  & Minimal / precise       & Simplest correct solution; no extra code \\
7  & Edge-case defender      & Enumerate: empty, single, zero, negative, max \\
8  & Algorithm-first         & Choose data structure before writing a line \\
9  & Output-format guard     & Verify return type (int/list/str/bool/tuple) \\
10 & Pseudocode-then-code    & English sketch first, then translate \\
11 & Constraint-aware        & Match time complexity to stated constraints \\
12 & Defensive / robust      & Assume worst-case valid input \\
13 & Readability-first       & Meaningful names; correct beats clever \\
14 & Verify-against-examples & Mental verification before finalising \\
15 & Direct and focused      & Transformation-first; no extra output \\
\bottomrule
\end{tabular}
\end{center}

The file ends with \fn{assert len(SEED\_PROMPTS) == 15} --- if a seed is accidentally
added or removed the program crashes immediately on import, before any API calls are made.

In \fn{ga.run()}, these 15 strings become \fn{Individual} objects:
\begin{lstlisting}[style=py]
population = [Individual(prompt=p) for p in seed_prompts]
# Each Individual: prompt=<seed>, fitness=-1.0, uid=<random 8-char hex>,
#                  parent_a="", parent_b="", operators=""
\end{lstlisting}


% ==========================================================================
\chapter{\pyf{dataset.py} --- Loading \& Splitting Problems}
% ==========================================================================

\section{\fn{load\_problems(apps\_train\_dir)} $\to$ \fn{list[dict]}}

\begin{spec}{load\_problems(apps\_train\_dir) -> list[dict]}
Called by: main.main() [main.py:75] \\
Calls: Path.iterdir(), json.loads(), Path.read\_text()
\end{spec}

Iterates every subdirectory of \pyf{APPS/train/}.  Each subdirectory is one problem.
Three files are required: \fn{metadata.json}, \fn{input\_output.json}, \fn{question.txt}.
A five-condition filter keeps only problems that are:
(1)~\fn{difficulty == "introductory"},
(2)~have a \fn{fn\_name} field,
(3)~sourced from \fn{codewars.com},
(4)~have non-empty \fn{inputs}, and
(5)~have non-empty \fn{outputs}.

\textbf{Why Codewars only?}  Codewars problems use standalone functions
(\fn{def grader(score): return ...}), giving a consistent I/O format.  LeetCode uses
\fn{class Solution} methods which complicates code extraction.  Inputs are parsed Python
argument lists; outputs are \fn{[return\_value]} (single-element list).

Each passing problem produces:
\begin{lstlisting}[style=py]
{
    "id":       "3715",
    "question": "Complete the function that takes an array of words...",
    "fn_name":  "nth_char",
    "inputs":   [ [["yoda","best","has"]], [["a","bc","def"]] ],
    "outputs":  [ ["oes"], ["abd"] ]
    # inputs[i] = list of args  => fn_name(*inputs[i])
    # outputs[i] = [return_value]  => [result] == outputs[i]
}
\end{lstlisting}

\section{\fn{split(problems, evolution\_size, holdout\_size, seed=42)} $\to$ \fn{(list, list, list)}}

\begin{spec}{split(...) -> (evolution[100], holdout[200], rest)}
Called by: main.main() [main.py:82] \\
Calls: random.Random(seed), list.copy(), rng.shuffle()
\end{spec}

\begin{lstlisting}[style=py]
rng = random.Random(seed)   # isolated RNG, does not affect global random state
shuffled = problems.copy()  # never modify the caller's list
rng.shuffle(shuffled)       # deterministic: seed 42 always gives the same order

evolution = shuffled[:100]         # the GA's training set
holdout   = shuffled[100:300]      # reserved for final generalisation test
rest      = shuffled[300:]         # unused buffer
\end{lstlisting}

The fixed seed is critical: if the 100 evolution problems changed between generations,
fitness scores from generation 3 and generation 7 would not be comparable.

\begin{concept}{Train / test split}
The GA "trains" on 100 problems (evolution pool).  The best evolved prompt is
then evaluated on 200 never-seen holdout problems.  If holdout fitness is much lower
than evolution fitness, the prompt has overfitted to the 100 training problems.
\end{concept}


% ==========================================================================
\chapter{\pyf{evaluator.py} --- Code Extraction \& Sandboxed Testing}
% ==========================================================================

\section{Overview}

Two responsibilities: (1) strip Python code out of an LLM's raw text response, and
(2) run that code in an isolated subprocess and return a partial-credit score.

\section{\fn{extract\_code(response: str)} $\to$ \fn{str}}

\begin{spec}{extract\_code(response) -> str}
Called by: ga.compute\_fitness() [ga.py:68]; main.evaluate\_on\_holdout() [main.py:54] \\
Calls: re.search() (at most twice)
\end{spec}

Tries three strategies in order, returning on first success:

\begin{lstlisting}[style=py]
# 1. Tagged python fence
m = re.search(r"```python\s*\n([\s\S]*?)```", response)
if m: return m.group(1).strip()

# 2. Any fence
m = re.search(r"```\s*\n?([\s\S]*?)```", response)
if m: return m.group(1).strip()

# 3. Assume raw code
return response.strip()
\end{lstlisting}

Example: input \fn{"Here's my answer:\textbackslash n```python\textbackslash ndef nth\_char(words):\textbackslash n    ...\textbackslash n```"} $\to$ output \fn{"def nth\_char(words):\textbackslash n    ..."}.

\section{\fn{has\_function(code: str, fn\_name: str)} $\to$ \fn{bool}}

\begin{spec}{has\_function(code, fn\_name) -> bool}
Called by: run\_tests() [evaluator.py:97] \\
Calls: re.search(), re.escape()
\end{spec}

\begin{lstlisting}[style=py]
return bool(re.search(rf"\bdef\s+{re.escape(fn_name)}\s*\(", code))
# \b = word boundary, \s+ = spaces, \s* = optional space before (
# Guards against wrong function name (NameError), wrong class (method).
# If False, run_tests() returns 0.0 immediately without launching subprocess.
\end{lstlisting}

\section{\fn{run\_tests(code, fn\_name, inputs, outputs, timeout=5)} $\to$ \fn{float}}

\begin{spec}{run\_tests(...) -> float in [0.0, 1.0]}
Called by: ga.compute\_fitness() [ga.py:69]; main.evaluate\_on\_holdout() [main.py:56] \\
Calls: has\_function(), \_build\_script(), subprocess.run()
\end{spec}

\textbf{How it works end-to-end:}

\begin{lstlisting}[style=chain]
run_tests(code, fn_name, inputs, outputs, timeout=5)
  |
  +-- if not code or not has_function(code, fn_name):
  |       return 0.0    # early-out: no code or wrong function name
  |
  +-- script = _build_script(code, fn_name, inputs, outputs)
  |       Produces a complete Python script (see below)
  |
  +-- proc = subprocess.run(
  |       [sys.executable, "-c", script],
  |       capture_output=True, text=True, timeout=timeout)
  |       # Fully isolated process; if it crashes, GA is unaffected
  |
  +-- results = [int(line[len("__R__:"):])
  |               for line in proc.stdout.splitlines()
  |               if line.startswith("__R__:")]
  |       # Only sentinel lines are parsed; stray print() calls are ignored
  |
  +-- if not results: return 0.0   # script crashed before any test ran
  +-- return sum(results) / len(inputs)   # partial credit
        # denominator = TOTAL tests, not just those that produced output
        # e.g. 3 pass out of 4 = 0.75 even if script crashed mid-way
\end{lstlisting}

\textbf{The test harness script} assembled by \fn{\_build\_script()} looks like this
for an \fn{nth\_char} problem:

\begin{lstlisting}[style=py, caption={Script produced by \_build\_script() -- executed in subprocess}]
import sys
sys.set_int_max_str_digits(1_000_000)

def nth_char(words):               # <-- LLM-generated code pasted verbatim
    return ''.join(word[i] if i < len(word) else '#'
                   for i, word in enumerate(words, 1))

def __run__(inp, expected):
    try:
        result = nth_char(*inp)    # call the generated function
        # outputs are stored as [return_value], so wrap result in list:
        sys.stdout.write("__R__:" + ("1" if [result] == expected else "0") + "\n")
        sys.stdout.flush()
    except Exception:              # catches NameError, TypeError, etc.
        sys.stdout.write("__R__:0\n")
        sys.stdout.flush()

__run__([["yoda","best","has"]], ["oes"])   # one line per test case
__run__([["a","bc","def"]], ["abd"])
\end{lstlisting}

\begin{concept}{Why \fn{[result] == expected}?}
APPS Codewars outputs are stored as \fn{[return\_value]}.  Wrapping the actual
result in \fn{[result]} makes the comparison format match:
\fn{["oes"] == ["oes"]} is \fn{True}; \fn{"oes" == ["oes"]} would always be \fn{False}.
\end{concept}

\begin{warn}{Why a subprocess?}
Generated code is untrusted.  Direct import could crash the GA process (via
\fn{sys.exit()} or infinite loop with no timeout).  The subprocess is killed after
5 seconds; the GA continues regardless.
\end{warn}

\begin{warn}{Division by \fn{len(inputs)}, not \fn{len(results)}}
If generated code crashes after 2 of 4 tests, only 2 \fn{\_\_R\_\_:} lines appear.
But the denominator is still 4 (total tests), not 2 (completed tests).  This fairly
penalises code that fails partway through.
\end{warn}

\chapter{\pyf{cost\_tracker.py} --- API Cost Logging}
% ==========================================================================

\section{Purpose \& Location}

Every Claude API call has a dollar cost.  This module logs token counts, computes
costs from official Anthropic pricing, and writes a JSON breakdown to
\fn{Code/Costs/cost\_TIMESTAMP.json} after each run.

\begin{lstlisting}[style=py]
COSTS_DIR = Path(__file__).parent / "Costs"   # Code/Costs/
\end{lstlisting}

\section{\fn{CallRecord} --- One API Call's Data}

\begin{spec}{@dataclass CallRecord}
Created by: CostTracker.record() [cost\_tracker.py:73] \\
Stores: phase (str), model (str), input\_tokens (int), output\_tokens (int) \\
Computes on-the-fly: input\_cost, output\_cost, total\_cost (via PRICING dict)
\end{spec}

\begin{lstlisting}[style=py]
PRICING = {   # (input_$/MTok, output_$/MTok)
    "claude-haiku-4-5-20251001": (1.00,  5.00),
    "claude-sonnet-4-6":         (3.00, 15.00),
    ...
}

@property
def total_cost(self) -> float:
    rate_in, rate_out = PRICING.get(self.model, (0, 0))
    return (self.input_tokens * rate_in
          + self.output_tokens * rate_out) / 1_000_000
# Example: 287 input tokens + 52 output tokens on Haiku 4.5:
#   = (287*1.00 + 52*5.00) / 1e6 = $0.000547 per call
\end{lstlisting}

\section{\fn{CostTracker} --- The Singleton Accumulator}

\begin{spec}{class CostTracker}
Module-level singleton: tracker = CostTracker() [line 197] \\
Imported by llm.py as: from cost\_tracker import tracker
\end{spec}

\begin{lstlisting}[style=chain]
CostTracker.record(phase, model, input_tokens, output_tokens)
  Called by: llm._call() after every successful API response
  Action: appends CallRecord to self.calls (in-memory log)

CostTracker._by_phase() -> dict
  Groups self.calls by phase ("fitness", "crossover", "mutate_inject", ...)
  Returns totals per phase: calls, tokens, cost

CostTracker._by_model() -> dict
  Groups self.calls by model ID
  Returns totals per model

CostTracker.flush(extra=None) -> Path
  Called by: main.main() [main.py:154]  -- once per run, at the very end
  Action:
    - COSTS_DIR.mkdir(parents=True, exist_ok=True)   # create if needed
    - assembles JSON report: totals + _by_phase() + _by_model() + pricing_used
    - writes Code/Costs/cost_TIMESTAMP.json
    - returns Path of the written file
\end{lstlisting}

\begin{concept}{Singleton pattern}
Python caches imported modules.  \fn{from cost\_tracker import tracker} in both
\pyf{llm.py} and \pyf{main.py} gets the \emph{exact same} \fn{CostTracker} instance.
So every \fn{\_call()} adds to the same list, and \fn{flush()} sees the complete
history for the whole run.
\end{concept}


% ==========================================================================
\chapter{\pyf{llm.py} --- All Claude API Calls}
% ==========================================================================

\section{Overview}

The sole interface to the Anthropic API.  All five public functions route through the
private \fn{\_call()} wrapper, which handles authentication, retries, and cost logging.

\begin{lstlisting}[style=chain]
Public API:
  generate_code(system_prompt, question, fn_name)  -> str  [raw LLM text]
  crossover(prompt_a, prompt_b)                    -> str  [merged prompt]
  mutate_inject(prompt)                            -> str  [prompt + strategy]
  mutate_delete(prompt)                            -> str  [prompt - sentence]
  mutate_rephrase(prompt)                          -> str  [prompt rephrased]

All call through:
  _call(phase, model, temperature, max_tokens, system, user) -> str
    which calls _get_client() once (lazy init) then client.messages.create(...)
\end{lstlisting}

\section{\fn{\_call()} --- The Core Wrapper}

\begin{spec}{\_call(phase, model, temperature, max\_tokens, system, user, retries=3) -> str}
Called by: all five public functions \\
Calls: \_get\_client(), client.messages.create(), tracker.record(), time.sleep()
\end{spec}

\begin{lstlisting}[style=py, caption={\_call() -- retry and logging logic}]
def _call(phase, model, temperature, max_tokens, system, user, retries=3):
    client = _get_client()   # returns cached anthropic.Anthropic(api_key=...)
    for attempt in range(retries):
        try:
            resp = client.messages.create(
                model=model, max_tokens=max_tokens,
                temperature=temperature, system=system,
                messages=[{"role": "user", "content": user}],
            )
            tracker.record(phase=phase, model=model,  # log tokens + cost
                input_tokens=resp.usage.input_tokens,
                output_tokens=resp.usage.output_tokens)
            return resp.content[0].text               # the assistant's response

        except anthropic.RateLimitError:
            wait = 2 ** attempt * 5   # 5s, 10s, 20s (exponential backoff)
            time.sleep(wait)
        except anthropic.APIError as e:
            if attempt == retries - 1: return ""      # give up; return empty
            time.sleep(2)
    return ""   # all retries exhausted
\end{lstlisting}

\begin{concept}{What happens when \fn{\_call} returns \fn{""}}
An empty response propagates as: \fn{extract\_code("")} $\to$ \fn{""}, then
\fn{run\_tests("", ...)} $\to$ 0.0 immediately (empty-code guard).
The problem scores 0; the run continues.  One failed API call does not crash the GA.
\end{concept}

\section{\fn{Individual} --- The Chromosome Dataclass}

\begin{lstlisting}[style=py]
@dataclass
class Individual:

\section{\fn{compute\_fitness(individual, problems)} $\to$ \fn{float}}

\begin{spec}{compute\_fitness(individual, problems) -> float in [0.0, 1.0]}
Called by: run() -- 15 times at gen 0; 13 times per generation (gens 1--20) \\
Calls: llm.generate\_code(), evaluator.extract\_code(), evaluator.run\_tests()
\end{spec}

\begin{lstlisting}[style=chain]
compute_fitness(individual, problems):
  total = 0.0
  for prob in problems:       # 100 iterations

    response = llm.generate_code(         [llm.py:77]
                   individual.prompt,
                   prob["question"],
                   prob["fn_name"])
               -> _call("fitness", Haiku, T=0.0, max=1024, ...)
               -> tracker.record(tokens)
               -> returns raw LLM text

    code = evaluator.extract_code(response)   [evaluator.py:24]
               -> returns code string

    score = evaluator.run_tests(              [evaluator.py:84]
                code, prob["fn_name"],
                prob["inputs"], prob["outputs"], timeout=5)
               -> has_function() check
               -> _build_script() assembles harness
               -> subprocess.run() executes
               -> parse __R__: lines
               -> returns float [0.0, 1.0]

    total += score

  return total / len(problems)   # mean partial credit across 100 problems
\end{lstlisting}

\begin{warn}{Expensive operation}
One call to \fn{compute\_fitness()} makes \textbf{100 Haiku API calls}.
At $\sim$\$0.0005 per call, evaluating one individual costs $\sim$\$0.05.
Evaluating all 275 individuals in a full run costs $\sim$\$13--25.
\end{warn}

\section{\fn{tournament\_select(population, rng)} $\to$ \fn{Individual}}

\begin{spec}{tournament\_select(population, rng) -> Individual}
Called by: run() -- twice per offspring [ga.py:199--200] \\
Calls: rng.sample(population, 3), max(..., key=fitness)
\end{spec}

\begin{lstlisting}[style=py]
contestants = rng.sample(population, config.TOURNAMENT_SIZE)  # 3 draws, no replace

# The fittest of 3 random draws becomes a parent.
# Weaker individuals can win if stronger ones are not drawn.
# This balances exploitation (best usually wins) with exploration.
\end{lstlisting}

\section{\fn{breed(parent\_a, parent\_b, rng)} $\to$ \fn{Individual}}

\begin{spec}{breed(parent\_a, parent\_b, rng) -> Individual (fitness=-1.0)}
Called by: run() [ga.py:201] -- 13 times per generation \\
Calls: llm.crossover() OR rng.choice(), then optionally llm.mutate\_inject/delete/rephrase()
\end{spec}

\begin{lstlisting}[style=chain]
breed(parent_a, parent_b, rng):

  [STEP 1 -- Crossover or copy]
  if rng.random() < 0.80:
      prompt = llm.crossover(parent_a.prompt, parent_b.prompt)
               -> _call("crossover", Sonnet, T=0.7, max=600, ...)
               -> returns merged prompt string
      ops = ["crossover"]
  else:
      prompt = rng.choice([parent_a.prompt, parent_b.prompt])
      ops = ["copy"]

  [STEP 2 -- Inject? (independent roll)]
  if rng.random() < 0.25:
      prompt = llm.mutate_inject(prompt)
               -> _call("mutate_inject", Sonnet, T=0.7, max=600, ...)
      ops.append("inject")

  [STEP 3 -- Delete? (independent roll)]
  if rng.random() < 0.15:
      prompt = llm.mutate_delete(prompt)   # may skip _call if 1 sentence
      ops.append("delete")

  [STEP 4 -- Rephrase? (independent roll)]
  if rng.random() < 0.10:
      prompt = llm.mutate_rephrase(prompt)
               -> _call("mutate_rephrase", Sonnet, T=0.7, max=600, ...)
      ops.append("rephrase")

  return Individual(
      prompt=prompt,
      parent_a=parent_a.uid,          # genealogy tracking
      parent_b=parent_b.uid,
      operators="+".join(ops),        # e.g. "crossover+inject" or "copy"
  )
  # fitness = -1.0 (sentinel; run() calls compute_fitness() separately)
\end{lstlisting}

\section{\fn{run(seed\_prompts, problems, on\_generation=None)}}

\begin{spec}{run(...) -> (best\_Individual, history: list[dict])}
Called by: main.main() [main.py:120] \\
Calls: compute\_fitness(), tournament\_select(), breed(), on\_generation callback
\end{spec}

\begin{lstlisting}[style=chain]
run(seed_prompts, problems, on_generation):

  rng = random.Random(42)    # seeded; reproducible across runs
  population = [Individual(prompt=p) for p in seed_prompts]  # 15 individuals

  [GEN 0] for each ind in population:
      ind.fitness = compute_fitness(ind, problems)    # 15 x 100 = 1500 API calls

  [GENS 1..20] for gen in range(1, 21):

      population.sort(by fitness, descending)

      stats = {generation: gen-1, best, mean, worst, best_uid, best_prompt}
      history.append(stats)       # records state of PREVIOUS generation

      if on_generation:
          on_generation(gen-1, population, stats)
          # = main.save_checkpoint(run_dir, gen-1, pop, stats)
          # Writes: results/run_*/generation_NN.json

      new_pop = population[:2]    # ELITISM: top 2 carried unchanged

      while len(new_pop) < 15:    # breed 13 new offspring
          p1 = tournament_select(population, rng)
          p2 = tournament_select(population, rng)
          child = breed(p1, p2, rng)               # 0-4 Sonnet calls
          new_pop.append(child)

      for ind in new_pop[2:]:     # evaluate only the 13 NEW offspring
          ind.fitness = compute_fitness(ind, problems)   # 13 x 100 = 1300 calls

      population = new_pop

  [FINAL]
  population.sort(by fitness, descending)
  history.append({generation: 20, ...})   # 21st record

  return (population[0], history)         # best prompt + 21-entry history
\end{lstlisting}

\begin{concept}{Generational replacement}
The entire population is replaced each generation (except the 2 elites).  This is
\textbf{full generational replacement}: only the top 2 are guaranteed to survive.
It creates strong selection pressure and fast convergence at the cost of some diversity.
\end{concept}

\begin{concept}{History records}
\fn{history} has 21 entries (generations 0--20).  Each entry has:
\fn{generation}, \fn{best}, \fn{mean}, \fn{worst}, \fn{best\_uid}, \fn{best\_prompt}.
Generations 0--19 are recorded at the \emph{start} of the next breeding step;
generation 20 is recorded after the loop ends.
\end{concept}

\chapter*{Glossary}
\addcontentsline{toc}{chapter}{Glossary}
% ==========================================================================

\begin{description}[style=nextline, leftmargin=2.8cm, itemsep=2pt]

\item[Chromosome] A candidate solution in a GA.  Here: a system prompt string,
  stored as an \fn{Individual} object.

\item[Crossover] Genetic operator combining two parents into one offspring.  Here:
  an LLM semantically blends two parent prompts.  Fires with probability 0.8.

\item[Elitism] Copying the top $k$ individuals unchanged into the next generation
  (\fn{ELITISM\_COUNT=2}).  Prevents the best solution from ever being lost.

\item[Evolution pool] The fixed 100 APPS problems used as the fitness benchmark.
  Chosen by deterministic shuffle with seed 42.

\item[Exponential backoff] Waiting 5s, 10s, 20s between retries on rate-limit errors
  (\fn{wait = 2**attempt * 5} in \fn{\_call()}).

\item[Fitness] Mean partial-credit score across 100 problems.  Range: [0.0, 1.0].
  Computed by \fn{ga.compute\_fitness()}.

\item[\fn{fn\_name}] The Python function name the LLM must write, from each problem's
  \fn{input\_output.json}.

\item[Generational replacement] The entire population is replaced each generation
  (except elites), creating strong selection pressure.

\item[Held-out set] 200 problems never seen during evolution.  Used to test
  generalisation of the final evolved prompt.

\item[Individual] One GA population member: a prompt with fitness, uid, parent uids,
  and operator history.

\item[Lazy init] \fn{\_get\_client()} only creates the \fn{anthropic.Anthropic} object
  on first use, not at module import time.

\item[Meta-prompt] A prompt sent to the Optimizer LLM instructing it to modify another
  prompt (used in crossover and all three mutation operators).

\item[Mutation] Modifying a single chromosome.  Three independent types: inject (25\%),
  delete (15\%), rephrase (10\%).

\item[Partial credit] Score = (tests passed) / (total tests).  Provides a smooth
  gradient vs.\ binary pass/fail.

\item[Sentinel] The \fn{\_\_R\_\_:} prefix tags test results in subprocess stdout,
  separating them from any stray \fn{print()} calls in generated code.

\item[Singleton] The \fn{tracker = CostTracker()} instance at module level is shared
  by all importers (\pyf{llm.py} and \pyf{main.py}).

\item[System prompt] English instructions sent before the user message, shaping LLM
  behaviour.  The GA chromosome.

\item[Temperature] LLM randomness parameter.  0.0 = deterministic (target model);
  0.7 = creative (optimizer model).

\item[Tournament selection] Sample $k=3$ individuals at random; return the fittest.
  Balances exploitation with exploration.

\item[Two-model architecture] Haiku 4.5 (cheap, deterministic) is optimised;
  Sonnet 4.6 (capable, creative) performs the genetic operators.

\end{description}
