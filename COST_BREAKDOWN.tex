\chapter{Token Measurements}
% ======================================================================

Token sizes were \textbf{measured from real APPS data}, not estimated. The evolution pool of 100 problems was loaded and measured; solution files from the dataset were used to gauge output length.

\begin{table}[h]
\centering
\begin{tabular}{@{}lrrl@{}}
\toprule
\textbf{Call type} & {\textbf{Input (tokens)}} & {\textbf{Output (tokens)}} & \textbf{How measured} \\
\midrule
Fitness (code gen)   & 280  & 138  & system prompt $\sim$48 + question $\sim$232 \\
Crossover            & 350  & 100  & meta-prompt $\sim$150 + two parents $\sim$100 each \\
Mutation (any type)  & 200  & 100  & meta-prompt $\sim$100 + one parent $\sim$100 \\
\bottomrule
\end{tabular}
\caption{Measured token sizes per API call. Source: real APPS evolution pool.}
\label{tab:tokens}
\end{table}

\subsection*{Measurement methodology}

\begin{itemize}[nosep]
  \item \textbf{Question length}: Average of \texttt{len(question)} for 100 sampled APPS problems, divided by 4 (chars-to-tokens approximation). Result: 876 chars $\approx$ 219 tokens. Plus system prompt ($\sim$48 tokens) and function-name suffix ($\sim$13 tokens) = 280 total input.
  \item \textbf{Output length}: Average of \texttt{len(solution)} from 30 sampled \texttt{solutions.json} files, divided by 4. Result: 552 chars $\approx$ 138 tokens.
  \item \textbf{Question range}: 164 -- 5{,}400 chars (41 -- 1{,}350 tokens). The 280-token average accounts for the full distribution.
\end{itemize}

\textbf{Total Sonnet} & & \textbf{338} & \\

\bottomrule
\end{tabular}
\caption{API call counts by phase.}
\label{tab:calls}
\end{table}

\subsection*{Key assumptions}

\begin{itemize}[nosep]
  \item Population size = 15, elitism = 2 $\Rightarrow$ 13 new offspring per generation.
  \item Total offspring across 20 generations = $13 \times 20 = 260$.
  \item Crossover rate = 0.80, so $\sim$208 crossover calls.
  \item Mutation probabilities are independent per offspring: inject~0.25, delete~0.15, rephrase~0.10.
  \item Elite individuals are \emph{not} re-evaluated (fitness carried from previous generation).
\end{itemize}

\chapter{Optimisation Options}
% ======================================================================

\section{Three Levers to Reduce Cost}

\begin{enumerate}
  \item \textbf{Reduce evolution pool size} (100 $\rightarrow$ 50 or 75 problems): Halves or reduces fitness calls proportionally. GAAPO~\cite{gaapo} showed that bandit-based selection with 50 samples matched full evaluation quality.
  \item \textbf{Reduce generations} (20 $\rightarrow$ 10 or 15): GAAPO found that larger populations with fewer generations outperformed smaller populations with more generations~\cite{gaapo}.
  \item \textbf{Batch API} (50\% off Haiku): Processes calls asynchronously. Fitness evaluation is embarrassingly parallel --- all 100 problems per individual are independent.
\end{enumerate}

\chapter{Runtime Cost Tracking}
% ======================================================================

The \texttt{cost\_tracker.py} module (see Code Explanation document) automatically logs every API call's exact token count and computes the real cost. After each run, a JSON file is saved to \texttt{Costs/cost\_<timestamp>.json} with this structure:

\begin{verbatim}
{
  "grand_total_usd": 27.67,
  "breakdown_by_phase": {
    "fitness":        { "calls": 27700, "total_cost_usd": 26.87 },
    "crossover":      { "calls": 208,   "total_cost_usd": 0.53 },
    "mutate_inject":  { "calls": 65,    "total_cost_usd": 0.14 },
    "mutate_delete":  { "calls": 39,    "total_cost_usd": 0.08 },
    "mutate_rephrase":{ "calls": 26,    "total_cost_usd": 0.06 }
  },
  "breakdown_by_model": {
    "claude-haiku-4-5-20251001": { "total_cost_usd": 26.87 },
    "claude-sonnet-4-6":         { "total_cost_usd": 0.80 }
  }
}
\end{verbatim}

Each run creates a \textbf{new file} --- they never overwrite. This gives a full cost history across all experiments. The \texttt{grand\_total\_usd} figure in each file is the \emph{actual measured cost} (from real API token counts), not an estimate.
